# 像素小人电子宠物网页游戏：项目总规划 v0.1

## 1. 项目定位

本项目是一个电脑网页优先的像素风电子宠物/小人生活游戏。玩家照看住在网页小屋里的像素小人，通过点击房间中的家具和物件触发互动，观察小人的状态、作息和情绪变化。

核心气质：

- 像素风
- 治愈、安静、温柔
- 慢节奏、低压力
- 现实时间驱动
- 以观察和陪伴为主，而不是任务压迫

第一版不追求复杂玩法，而是优先做出一个稳定、清晰、可扩展的生活原型：打开网页后，玩家能看到一个小人在房间中生活，切换房间，点击家具照顾小人，状态会变化，并且数据会在本设备保存。

## 2. 第一版目标

第一版目标是完成一个可玩的 MVP：

1. 一个像素小人角色出现在房间里。
2. 系统预留未来支持多只小人，但第一版画面只展示一只。
3. 提供四个独立房间：客厅、卧室、卫生间、厨房。
4. 通过点击家具/场景物件触发基础互动。
5. 支持进食、洗澡、睡觉、玩耍四种基础互动。
6. 使用状态条直观显示饱腹、清洁、精力、心情、亲密度。
7. 按现实时间显示清晨、白天、傍晚、夜晚。
8. 按现实月份显示春、夏、秋、冬。
9. 使用游戏内模拟天气，不接真实天气服务。
10. 使用本地保存，不做后端、账号、云同步。
11. 电脑网页优先，手机端只保证基础可访问，不作为第一版主要体验目标。

## 3. 第一版不做的内容

为了控制范围，第一版暂不包含：

- 后端服务
- 用户账号
- 云存档
- 真实天气接口
- 多设备同步
- 多个小人同时出现在画面中
- 小人之间的完整互动系统
- 家具解锁系统
- 装扮系统
- 商店/货币系统
- 任务系统
- 室外场景
- 节日活动
- 自由摆放家具
- 复杂路径移动或自由地图探索

这些内容都可以作为后续扩展，但不进入第一版核心范围。

## 4. 角色设计原则

角色不是动物，而是一个可爱的像素小人。小人不直接说话，不使用对话气泡表达台词；它主要通过动作、表情、状态条变化和少量旁白反馈表达状态。

第一版只展示一只小人，但设计上要预留未来支持多只小人。每个小人都应作为独立角色设计，而不是只替换外观。

### 4.1 小人设计模板

后续每个小人都应包含以下设定：

- 名称
- 外形轮廓
- 身体比例
- 发型/头部特征
- 默认服装
- 主色和辅助色
- 视觉锚点
- 性格关键词
- 喜欢的互动或物品
- 不喜欢的互动或状态
- 饥饿时的表现
- 困倦时的表现
- 脏了时的表现
- 开心时的表现
- 低落时的表现
- 待机动作
- 进食动作
- 洗澡动作
- 睡觉动作
- 玩耍动作
- 与其他小人的互动风格
- 亲密度提升后的变化
- 后续可解锁动作
- 后续可扩展装扮方向

### 4.2 性格系统原则

性格不是后加的数值系统，而是每个小人的角色核心。设计每个小人时，应同时设计它的情绪展示逻辑和互动动作。

例如，同样是“开心”：

- 安静型小人可能只是轻轻晃动、脸红。
- 活泼型小人可能跳起来或转圈。
- 懒散型小人可能伸懒腰或趴在地毯上。
- 害羞型小人可能低头、遮脸、微微摇晃。

同样是“进食”：

- 贪吃型小人吃得快，吃完拍拍肚子。
- 文静型小人慢慢吃，吃完轻轻点头。
- 笨拙型小人可能掉一点食物，吃完不好意思。

第一版可以先只实现一个小人的一套动作，但数据和代码组织应避免把动作写死成所有小人共用。

## 5. 房间设计

第一版包含四个独立房间。房间之间通过场景切换进入，不做一个连续的大地图。

### 5.1 客厅

定位：休息、玩耍、日常观察的主要空间。

核心物件：

- 玩具箱或玩具
- 沙发
- 地毯
- 窗户
- 门/房间切换入口

主要互动：玩耍。

### 5.2 卧室

定位：睡觉、恢复精力、安静氛围。

核心物件：

- 床
- 床头柜
- 衣柜
- 台灯
- 窗户

主要互动：睡觉。

### 5.3 卫生间

定位：清洁、洗澡。

核心物件：

- 浴缸或淋浴区
- 洗手池
- 镜子
- 毛巾

主要互动：洗澡。

### 5.4 厨房

定位：进食。

核心物件：

- 餐桌或小餐台
- 冰箱
- 灶台
- 食物/餐具

主要互动：进食。

## 6. 互动设计

第一版以点击家具/场景物件作为主要触发方式，不在底部放一排传统功能按钮。

基础互动：

| 互动 | 所在房间 | 触发物件 | 主要影响 |
| --- | --- | --- | --- |
| 进食 | 厨房 | 餐桌/食物/冰箱 | 饱腹上升，心情小幅上升，清洁可能小幅下降 |
| 洗澡 | 卫生间 | 浴缸/淋浴 | 清洁上升，心情小幅上升，精力小幅下降 |
| 睡觉 | 卧室 | 床 | 精力上升，饱腹缓慢下降 |
| 玩耍 | 客厅 | 玩具箱/玩具 | 心情上升，精力下降，饱腹下降，清洁小幅下降 |

互动流程建议：

1. 玩家点击可互动家具。
2. 如果当前小人可行动，则进入对应互动状态。
3. 播放简化动作或占位动画。
4. 调整状态条数值。
5. 更新亲密度或互动记录。
6. 保存数据。
7. 小人回到待机状态。

后续可以增加“小人先走到家具旁边再互动”，但第一版不强制要求复杂移动。

## 7. 状态系统

第一版使用状态条，而不是隐藏图标或让玩家猜测。状态条直观显示小人状态，但 UI 风格应保持柔和，避免变成战斗游戏或管理后台。

### 7.1 状态条类型

- 饱腹
- 清洁
- 精力
- 心情
- 亲密度

### 7.2 数值原则

内部可以使用 0-100 的数值，界面上显示为条形进度。第一版不需要显示精确数字。

建议：

- 饱腹低：小人更容易表现出饿、动作变慢。
- 清洁低：小人可能出现脏污占位效果或低落状态。
- 精力低：小人容易困倦、待机动作变慢。
- 心情低：小人动作减少、表情低落。
- 亲密度：长期缓慢增长，不应频繁下降。

### 7.3 低压力原则

游戏不应惩罚玩家长时间离线。第一版不设计死亡、离家出走或严重惩罚。

玩家隔一段时间回来时，小人可以状态变低、房间显得安静或落灰，但应能通过照顾恢复。

## 8. 时间、季节与天气

### 8.1 时间

时间按照现实时间划分：

- 清晨
- 白天
- 傍晚
- 夜晚

具体时段可在实现时调整。例如：

- 05:00-08:59：清晨
- 09:00-16:59：白天
- 17:00-19:59：傍晚
- 20:00-04:59：夜晚

时间影响：

- 房间光照
- 窗外颜色
- 小人默认行为倾向
- 精力恢复/下降节奏

### 8.2 季节

季节按照现实月份：

- 春：3-5 月
- 夏：6-8 月
- 秋：9-11 月
- 冬：12-2 月

季节主要通过窗外景色和少量色调变化体现。第一版不需要为每个季节绘制完整室内背景。

### 8.3 天气

天气使用游戏内模拟，不接真实天气接口。天气可以按日期生成，保证同一天内相对稳定。

基础天气：

- 晴
- 多云/阴
- 雨
- 雪，主要在冬季出现

天气主要通过窗外、光照层和简单特效体现。

## 9. 网络依赖策略

第一版应尽量不依赖外部互联网服务。游戏可以作为一个本地优先的网页体验运行：核心玩法、状态计算、时间判断、季节判断、天气模拟和存档都应在浏览器端完成。

第一版不接入：

- 真实天气 API
- 用户账号服务
- 云存档服务
- 在线排行榜或社交服务
- 远程素材动态加载

第一版允许使用：

- 本地项目内置素材
- 浏览器本地时间
- 浏览器本地存储
- 游戏内按日期生成的模拟天气

这样可以降低开发复杂度，避免网络权限、接口稳定性、接口费用和隐私授权问题。后续如果要加入真实天气或云存档，应作为可选增强功能，而不是第一版核心依赖。

## 10. 本地保存

第一版不做后端，只在当前设备浏览器中保存数据。

需要保存：

- 当前小人数据
- 状态条数值
- 亲密度
- 当前房间
- 上次访问时间
- 当日天气
- 最近互动记录
- 后续预留的多小人数据结构

限制说明：

- 换设备不会同步。
- 换浏览器可能没有存档。
- 清空浏览器数据可能丢失存档。

这些限制在第一版可以接受。

## 11. 美术策略

为了避免画两遍，第一阶段不绘制大量临时美术。

### 11.1 第一阶段功能原型

使用占位素材验证功能：

- 小人可用简单轮廓或占位像素块。
- 房间可用色块、文字和简单家具形状。
- 可互动区域应清楚标注。
- 状态条可以先接近正式 UI，因为后续改动成本较低。

功能原型重点验证：

- 房间切换是否清楚
- 点击家具是否自然
- 状态变化是否合理
- 时间/季节/天气逻辑是否工作
- 本地保存是否可靠

### 11.2 正式视觉样张

在功能原型之外，可单独制作少量正式视觉样张用于定调：

- 一个正式小人的静态设定
- 一个正式客厅场景
- 一套状态条 UI 风格
- 一个白天效果
- 一个夜晚或傍晚效果

样张确认后，再批量制作正式房间、动作、天气和季节素材。

### 11.3 正式素材制作顺序

建议顺序：

1. 小人基础设定和站姿。
2. 小人待机动画。
3. 客厅正式样板。
4. 状态条 UI 风格。
5. 四个基础互动动作。
6. 厨房、卧室、卫生间。
7. 时间光照层。
8. 季节窗外层。
9. 天气特效层。

## 12. 开发阶段建议

### 阶段 1：结构原型

目标：完成可点击、可切换、可保存的基础体验。

内容：

- 基础页面布局
- 四个房间切换
- 一个占位小人
- 五个状态条
- 点击家具触发四个互动
- 基础状态变化
- 本地保存
- 现实时间判断
- 季节判断
- 模拟天气

### 阶段 2：生活感增强

目标：让小人更像在生活。

内容：

- 待机状态变化
- 状态低时的占位表现
- 简单互动动画
- 少量旁白提示
- 夜晚/傍晚光照层
- 雨/雪占位特效

### 阶段 3：正式视觉样张接入

目标：确认最终美术方向。

内容：

- 接入正式小人静态图或待机动画
- 接入正式客厅样张
- 调整角色尺寸和场景比例
- 调整状态条风格

### 阶段 4：正式 MVP 美术替换

目标：将占位内容替换为第一版正式素材。

内容：

- 四个正式房间
- 四个基础互动动画
- 时间光照效果
- 季节窗外变化
- 天气效果

### 阶段 5：扩展系统

目标：开始加入长期养成内容。

内容可包括：

- 多小人
- 小人之间互动
- 性格差异
- 家具解锁
- 装扮系统
- 更多房间或室外场景

## 13. 后续扩展方向

后续可以逐步扩展：

- 多个小人同时生活
- 小人之间的关系和互动
- 每个小人的独立性格与动作系统
- 家具收集和解锁
- 装扮系统
- 更多室内房间
- 花园、公园、商店等室外场景
- 季节限定装饰
- 节日事件
- 小型随机事件
- 更完整的日夜作息
- 真实天气可选项
- 账号和云存档

## 14. 数据结构预留

即使第一版只展示一只小人，也建议从一开始按可扩展数据组织，避免后续加入多小人、家具、性格差异时重写核心逻辑。

### 14.1 小人数据

每只小人建议预留：

- id
- 名称
- 外观资源
- 性格类型
- 状态数据
- 喜好数据
- 当前房间
- 当前动作
- 亲密度
- 解锁状态
- 特殊动画列表

### 14.2 房间数据

每个房间建议预留：

- 房间 id
- 房间名称
- 背景资源
- 家具列表
- 可互动点
- 小人可站立位置
- 时间段光照配置
- 季节装饰配置
- 天气表现配置

### 14.3 家具数据

每个家具建议预留：

- 家具 id
- 所属房间
- 类型
- 是否可点击
- 触发互动
- 互动位置
- 所需状态条件
- 动画资源
- 后续替换皮肤

### 14.4 互动数据

每个互动建议预留：

- 互动 id
- 互动类型
- 触发物件
- 持续时间
- 状态变化
- 小人动画
- 表情变化
- 旁白文本
- 冷却时间
- 性格差异表现

## 15. 验收标准

第一版完成后，应能回答以下问题：

- 玩家是否能不用底部功能按钮理解如何照顾小人？
- 四个房间是否各有清晰存在意义？
- 小人不说话时，是否仍然能通过动作和状态表达情绪？
- 状态条是否直观且不会破坏治愈氛围？
- 现实时间变化是否带来陪伴感？
- 季节和模拟天气是否增强小屋生活感？
- 本地保存是否足够支撑短期体验？
- 美术比例是否适合后续批量制作？
- 系统是否能自然扩展到多只小人？

## 16. 风险与控制策略

### 16.1 范围膨胀

风险：电子宠物很容易扩展出商店、任务、换装、多角色、剧情等内容，导致第一版迟迟无法完成。

控制策略：

- 第一版只围绕“照顾一只小人”。
- 所有新增功能都必须服务四个基础互动。
- 不提前做经济系统。
- 不提前做复杂收集。
- 不提前做多小人同屏。

### 16.2 美术返工

风险：过早绘制大量正式家具和动画，后续房间比例、角色尺寸、交互方式调整时会导致返工。

控制策略：

- 先用占位素材验证比例和交互。
- 只做少量正式样张定调。
- 批量素材制作前冻结画布尺寸、角色尺寸、家具比例和动画帧数标准。

### 16.3 角色表达不足

风险：小人不说话，如果动作和表情不足，可能显得无聊。

控制策略：

- 每个状态至少有一种视觉表现。
- 四个互动至少有开始、过程、结束反馈。
- 旁白只补充，不替代动画。
- 优先打磨待机、困倦、开心、低落等高频动作。

### 16.4 时间系统造成压力

风险：现实时间驱动可能让玩家感觉被催促或惩罚。

控制策略：

- 状态下降速度慢。
- 离线惩罚轻。
- 不设置死亡、离家出走或严重失败状态。
- 不使用强提醒。
- 以陪伴感为主，不以惩罚为主。

## 17. 当前决策摘要

当前已确定：

- 电脑网页优先。
- 第一版不做后端。
- 第一版用本地保存。
- 角色是小人，不是动物。
- 第一版只展示一只小人，但预留多只。
- 四个房间：客厅、卧室、卫生间、厨房。
- 互动通过点击家具触发。
- 基础互动：进食、洗澡、睡觉、玩耍。
- 状态条：饱腹、清洁、精力、心情、亲密度。
- 小人不说话。
- 时间按现实时间。
- 季节按现实月份。
- 天气游戏内模拟。
- 第一版不依赖外部互联网服务；真实天气、云存档等只作为后续可选增强。
- 美术第一阶段用占位，不画大量临时素材；另做少量正式样张定调。
- 性格系统应作为每个小人的角色设计核心，而不是后加数值。
